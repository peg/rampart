# Rampart 1.0-rc readiness gate

This is the bar for calling Rampart `1.0-rc` with a straight face.

It is not a marketing checklist. It is a trust checklist: every green box below should map to a command, a device run, a CI check, or an explicit release note tradeoff. It is also not an attacker cookbook: name the trust boundaries, prove the important behavior, and avoid publishing cute bypass recipes unless they are needed to explain a fix.

## Current blunt read

Rampart is close to `1.0-rc`, but the release candidate should not be cut until the live OpenClaw path has been proven on at least Linux and macOS.

What already looks real:

- core engine/proxy/bridge tests are passing
- OpenClaw native plugin path has deterministic smoke, approval, and degraded-mode coverage
- OpenClaw 2026.5.2 has first-class plugin approval forwarding, so Rampart can use OpenClaw as the single human-approval owner instead of creating a hidden second queue
- approval ownership is now single-owner: OpenClaw owns human approval UI; Rampart owns policy judgment, deny enforcement, audit, and durable allow-always persistence
- `rampart doctor` can verify plugin state, serve reachability, approval timeout alignment, and whether legacy OpenClaw exec-bundle hardening still applies to the installed build shape

What still needs explicit proof before `1.0-rc`:

- live OpenClaw approval cards on Linux and macOS
- live degraded-mode behavior on Linux and macOS
- install/repair/uninstall flow on macOS
- release notes that state fail-open/fail-closed behavior without overclaiming

## Initial bypass audit findings

### 1. Degraded mode is a real security boundary, not just an operational detail

OpenClaw plugin mode blocks sensitive tools when `rampart serve` is unavailable, while configured lower-risk tools can fail open. The preload/wrap paths have different defaults. `1.0-rc` needs to state this directly instead of implying every integration degrades the same way.

### 2. Interception coverage is stronger than semantic coverage

The native `before_tool_call` hook is the right OpenClaw architecture, but “all tools intercepted” must not read as “all tools are equally policy-rich under every failure mode.” The truthful statement is: all supported OpenClaw tool calls are intercepted; policy depth and degraded behavior depend on tool class and configuration.

### 3. Durable overrides are intentional trust boundaries

`user-overrides` and `auto-allowed` paths are useful operator-authorized escape hatches. They should be reviewed and documented as explicit trust boundaries, not treated as accidental bypasses or hidden implementation details.

### 4. Command and URL matching need adversarial regression proof

Command normalization is materially better, especially around shell wrappers. Keep a short adversarial matrix for wrapper chains, env prefixes, substitutions, malformed URLs, ports, and lookalike domains so the release claims stay evidence-backed.
